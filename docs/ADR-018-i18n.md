# ADR-018: Internationalization (pt-BR + en)

## Status: Accepted

## Context

The admin UI before FASE 13.3 shipped a mixed PT-BR + EN string pool:
some labels in Portuguese, some component text in English, error
messages in both. Two concrete problems:

1. **Multinational prospects** rejected the product after a 3-minute
   demo — the mixed-language UX read as "half-finished."
2. **Brazilian customers** encountered raw English in several places
   (notification templates, some status labels, Shield collectors),
   which made the product feel foreign despite being designed for
   the BR market.

Goal: ship a complete, consistent translation in both locales, with
a mechanism the team can extend per feature without risking drift.

## Decision

### Framework: `next-intl`

`next-intl` is the canonical i18n library for Next.js 14 App Router.
It provides:

- Server-component-aware translation resolution (`getTranslations` from
  `next-intl/server`)
- Client-component hook (`useTranslations`) that shares the same tree
- Build-time message loading (no runtime fetch)
- Locale detection from request headers, cookies, or URL prefix

### URL strategy: cookie-based, NO URL prefix

**Rejected:** `localePrefix: 'as-needed'` (pt-BR at `/`, en at `/en/...`).

**Accepted:** cookie-based locale. Same URL for both locales; the
`NEXT_LOCALE` cookie determines which language renders.

Rationale:

- The admin app is **not public** — no SEO, no crawling, no
  language-targeted landing pages. Localized URLs buy nothing here.
- The "as-needed" approach requires all 23 feature folders to move
  under `src/app/[locale]/`. That's a large, mechanical refactor with
  real regression risk (every import path, every test fixture, every
  link). The cookie approach leaves the file tree unchanged.
- Users expect the admin console to "just speak my language" — they
  don't link-share internal URLs that need locale semantics.

The trade-off: users on shared workstations must click the switcher
to change language (not automatic via URL). Acceptable for an
authenticated tool.

### Server-side locale resolution

`src/i18n/request.ts` implements the standard chain:

1. `NEXT_LOCALE` cookie (user's explicit choice — wins)
2. `Accept-Language` HTTP header (browser preference — degrades
   `pt` → `pt-BR`, `en-US` → `en`)
3. `DEFAULT_LOCALE` = `'pt-BR'` (matches target market)

Messages are imported dynamically by locale so only the active bundle
ships to the client. We do NOT use `LOCALE_COOKIE` ≠ `NEXT_LOCALE`
(the standard next.js cookie name) so that any hand-written middleware
or Next-native features that rely on the default name keep working.

### Message file structure

`admin-ui/src/i18n/messages/{pt-BR,en}.json` — one file per locale,
keyed by feature namespace (common, nav, chat, bias, icp, settings,
…). Each namespace is self-contained so a feature-owner can add/edit
keys without merge conflicts with unrelated work.

Key naming: camelCase on the leaves, dot-separated namespaces. Example:
`bias.verdict.pass`. Consumed as:

```tsx
const t = useTranslations('bias');
return <span>{t('verdict.pass')}</span>;
```

### Key parity guard

`src/__tests__/i18n.test.ts` loads both JSONs at test time and asserts:

- Every key path in pt-BR also exists in en (and vice-versa)
- No leaf value is empty or whitespace-only
- Both files have at least 150 keys (guards accidental truncation)

This catches the most common i18n regression at test time rather than
runtime: a developer adds a new feature, writes pt-BR keys, forgets
en. The test fails with the exact list of missing keys.

### Backend: per-org locale

Migration 083 adds `organizations.locale TEXT NOT NULL DEFAULT 'pt-BR'`
(CHECK constrained to `pt-BR` / `en`). This is the language the
**platform** uses when it initiates a message — notification templates,
email bodies, PDF reports, SIEM event labels. Per-user UI locale is
still the cookie.

Endpoint `PUT /v1/admin/settings/organization/locale` lets org admins
flip the org-wide preference without touching the broader settings
endpoint (back-compat).

## Trade-offs

- **No `/en/` URL paths.** If a customer later wants linkable English
  URLs, we'd switch to `localePrefix: 'as-needed'` and move pages
  under `[locale]/`. The migration is mechanical and the cookie
  approach is not a blocker for it.

- **Cookie persistence means a shared workstation shows the last
  picker's language.** Real risk is low — the admin app is
  authenticated per-user and each login has their own session.

- **Sidebar is fully localized; deep feature pages are localized
  incrementally.** We shipped the highest-impact surfaces (nav +
  locale switcher + auth strings). Each feature page can pick up
  `useTranslations('<namespace>')` at the owner's pace. The parity
  test only covers strings that exist in the message files, so
  unlocalized components stay on the current mixed-language defaults
  until they're touched.

- **Messages live in `admin-ui/src/i18n/messages/` but the parity test
  lives in the backend test suite.** Rationale: the backend test
  runner (vitest) runs as part of the regression pipeline. Duplicating
  the runner in admin-ui just for this one test isn't worth the
  complexity; a Node test with `readFileSync` covers it.

## Alternatives considered

- **`react-intl` / `formatjs`** — more mature in general, but doesn't
  integrate as cleanly with Next.js App Router's server components.
  next-intl is the framework's de-facto recommendation.

- **i18next + react-i18next** — powerful but heavier, and server-side
  rendering setup on App Router is awkward. Not worth the extra
  surface area when next-intl handles everything.

- **URL-prefix localization with middleware-driven rewrites.** Tried on
  paper: 23 page folders move, 50+ component imports churn, ~40 tests
  need fixture updates. Rejected on cost/benefit — no clear customer
  asking for localized URLs.

- **Single-file translation store (YAML/TOML).** JSON is native to
  the ecosystem (IDE tooling, lint rules, next-intl's loader).
  YAML would add a parser dep without fixing anything.

## Consequences

- Nav, locale switcher, auth strings render consistently in both
  languages out of the box.
- `useTranslations('<ns>')` is available to every component for
  incremental adoption.
- Key-parity regression test runs under vitest with 5 assertions —
  any feature adding strings in one language without the other
  fails CI.
- Org-level locale column enables localized backend emails/PDFs as a
  follow-up without another migration.
- Follow-ups: full localization of Playground, Dashboard, and form
  error messages across all remaining feature pages.
